## ⚠️ MAJOR FINDING: env version skew

**Local kaggle_environments: 1.28.1**

**pypi latest: 1.29.0**
**Kaggle ladder servers: at least 1.29.0; possibly newer**

### How it surfaced

Replay-based gate (3 top-player 2P replays from bowwowforeach,
sub-id 52318886) hit **84.7% match (260/307)** instead of the
expected ~100%. Categories that disqualified:
- 51 fleet-count-mismatch
- 51 fleet-id-set-mismatch
- 49 ship-count-off
- 22 ownership-flip

These are sweep/combat-related — same shape as pre-Phase-5 Day 5-7
mismatches.

### Investigation

Drilled into the first failing triple (episode 75930761 step 30):
- Fleet 8 at (81.704, 63.511), 26 ships, angle 0.6981
- Expected new pos (env's actual): (83.711, 65.195)
- Our sim's new pos: (83.711, 65.195) — IDENTICAL math
- Planet 14 at (82.734, 65.880) radius 1.6931
- Distance from planet 14 to fleet segment: 1.153
- 1.153 < 1.693 → our Phase 4 detects collision and destroys fleet 8
- Replay shows fleet 8 SURVIVING in state at step 31

**Crucial test**: injected the exact state into local env 1.28.1, ran
env.step() — env ALSO destroys fleet 8. So our sim is matching env
1.28.1 perfectly.

### Diff between 1.28.1 and 1.29.0 (orbit_wars.py only)

1. `generate_planets` and `generate_comet_paths` now take a seedable
   `rng` parameter (was module-level `random` calls).
2. **Q1/Q3 planet positions reflected across y=x diagonal**:
   - 1.28.1: `[id_counter, -1, x, y, ...]`
   - 1.29.0: `[id_counter, -1, y, x, ...]`
   - Comment explains: "Q1 and Q3 copies are reflected across the y=x
     diagonal so all 4 copies are 90° rotations of each other — every
     player sees the same game state rotated by their quadrant."
3. **"Phase 1.5 guaranteed orbiting group on y=x diagonal" REMOVED.**
4. Phase 4 collision check ORDER: planets first, then OOB+sun (was
   OOB+sun+planet). Logically equivalent for our specific failing case.

The (x,y) → (y,x) swap and Phase 1.5 removal are GAME GENERATION
changes that affect the initial planet layout. Per-step physics is
mathematically the same. The Phase 4 order swap doesn't affect
outcomes for fleets that hit planets without going OOB or hitting
sun (our case).

### Why fleet 8 survives in the replay despite seemingly equivalent physics

Hypotheses (NOT yet resolved):
- **A**: Kaggle ladder runs an even newer version (1.29.x or unreleased).
- **B**: There's a Phase 4/5 subtle difference between 1.28.1 and 1.29.0
  not caught in the order-of-checks diff.
- **C**: The replay's recorded state has subtle delta we're not extracting
  correctly (unlikely; we deepcopy).

Resolving requires either:
- Upgrade `kaggle_environments` to 1.29.0 and re-run the replay gate
- Run the same state injection on a 1.29.0 install
- Audit the full diff for non-obvious Phase 4 changes

### Implications (this is the important part)

**Our simulator IS byte-faithful to env 1.28.1.** Day 3-5/5-7/9-11
gates at 100% prove this conclusively.

**There is a continuous train/test gap at the env level itself**:
- Local diagnostics (`tools.diagnostic`, `tools.spot_check_4p`,
  `tools.trace_launch`) all use env 1.28.1.
- Local A/B winrate testing uses env 1.28.1.
- Modal CMA-ES tuner (`tools.modal_tuner`) uses env 1.28.1.
- Submitted agents are evaluated by Kaggle's server with env 1.29.0+.

This is on top of all the train/test gaps already documented in CLAUDE.md
(2P-vs-4P divergence, Python global random state consumption, etc.).

Per CLAUDE.md "do not pip install into the venv" — the upgrade
decision is the user's. Notes for the decision:
- `uv add kaggle-environments@1.29.0` would update uv.lock.
- Risk: 1.29.0 changes initial planet layout, so historical
  comparison runs (e.g., the v1.5G ladder benchmarks) become harder
  to reproduce locally. New runs would use new layouts.
- Benefit: closing the env version gap removes a class of
  train/test divergence; replay-based gate becomes meaningful;
  CMA-ES tuner trains against the same env Kaggle scores.
- Mitigation: pin both versions in a venv-per-task workflow if
  reproducibility of historical results matters more than alignment
  with current ladder.

## Notes for future-me

- **The replay download tool works around an SDK bug.** The Kaggle
  Python SDK's `competition_episode_replay()` uses chunked download
  but the replay endpoint sends `Content-Length` missing —
  `download_file()` crashes on `KeyError: 'content-length'`. Workaround:
  call the SDK's `get_episode_replay()` directly and read `response.content`.
- **Top players (this submission) have ~84 episodes**, not 74 as the
  notebook said. The notebook's number is from when it was captured.
- **Replays are ~1-1.5 MB each as JSON.** ~2000 episodes (the full
  notebook target) would be 2-3 GB of JSON. Don't download the full
  set without a reason; ~10 sample replays per top player is plenty
  for validation work.
- **Episode IDs are sortable; newer = larger.** Useful for time-series
  analysis (e.g., when an opponent's strategy changed).
