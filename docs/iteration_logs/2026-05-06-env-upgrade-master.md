## Discovery → fix sequence

1. **Day 9-11 gate landed at 100% on local env** but the new replay-based
   gate stuck at 84.7%. Investigation traced the divergence to **env
   version drift**: local was `kaggle-environments==1.28.1`, replays were
   generated by env 1.29.0+.
2. Upgraded to pypi `1.29.0` via `uv add`. Replay gate STILL failed at
   84.7% with the same mismatches. Drilled in: even injecting state into
   local env 1.29.0 destroyed fleet 8 (where the replay shows it surviving).
3. Checked GitHub master for `kaggle-environments`. Found a major
   architectural rewrite **not yet released to pypi**: a new helper
   `swept_pair_hit(A, B, P0, P1, r)` that does continuous-collision
   detection between two MOVING bodies in one tick. Old env's separate
   "Phase 4 (instantaneous fleet vs static planet) + Phase 5 sweep
   (rotating planet vs static fleet)" is replaced by a unified
   swept-pair check during fleet movement.
4. Installed `kaggle-environments` from `git+master` (commit 6458c31).
   Verified env now matches replay byte-for-byte for fleet 8.
5. Rewrote our simulator's Phase 4 + Phase 5 to mirror master env's
   swept-pair model. Updated all phase tests. All gates → 100%.

## Why fleet 8 survived in env master but died in env 1.29.0

The case that surfaced everything (episode 75930761 step 30):
- Fleet 8 segment: `(81.704, 63.511) → (83.711, 65.195)`
- Planet 14 starts at `(82.734, 65.880)` radius `1.693`
- Distance from planet 14 to fleet's segment = 1.153 → 1.153 < 1.693 → instantaneous check says COLLIDE.
- BUT planet 14 ROTATES to `(82.127, 67.073)` during the same tick.
- Swept-pair check: at no point in t∈[0,1] do the fleet trajectory and
  planet trajectory come within 1.693 of each other (planet moves AWAY
  from fleet while fleet moves SE).
- New env physics: NO collision. Fleet survives. ✓

The instantaneous check was a bug in old env's physics: it modeled the
planet as stationary during fleet movement, which it isn't. The new
physics fixes this.

## Code-level architectural change

**Before** (env 1.28.1 / 1.29.0 / our old sim):
```
Phase 4: for each fleet
  - update position
  - check OOB / sun / planet (instantaneous, planet at start position)
Phase 5: for each rotating planet
  - rotate to new position
  - sweep_fleets (planet's swept arc vs stationary fleet)
```

**After** (env master 6458c31 / our new sim):
```
Pre-compute planet paths (rotation + comet movement)
Phase 4: for each fleet
  - update position
  - check planets via swept_pair_hit(fleet_old, fleet_new, planet_old, planet_new, r)
  - check OOB / sun (planets first per env L511 comment)
Phase 5: just APPLY pre-computed planet positions; remove expired comets
```

The key insight from env master's design comment:
> "Compute each planet's end-of-tick position up front, so fleet movement
> can use a swept-pair (continuous) check that accounts for both objects
> moving in the same tick."
