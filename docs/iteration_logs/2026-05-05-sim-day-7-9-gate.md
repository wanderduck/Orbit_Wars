## Significant correctness fixes shipped in Day 7-9

### 1. Step-increment order fix (off-by-one in rotation)

**Before:** `step()` incremented `new_state.step` at the START. Phase 5 used `new_state.step = N+1` for rotation.
**After:** `step()` increments at the END. Phase 5 uses `new_state.step = N` (same as env's `obs0.step`).

Env source `step = get(obs0, "step", 1)` (env L555) reads the input step value, not the post-increment. Rotation formula is `initial_angle + ang_vel * step` — using N+1 instead of N would have produced rotated planets ahead of env by one tick of angular velocity. Did not catch it before because Day 3-5 / Day 5-7 gates filtered to scenarios where `state.fleets == []` or where rotation effects fell within state_diff tolerance for non-position fields.

### 2. Removed vestigial scaffolding

- `Simulator.skip_phase_5: bool = True` — was a Day 3-5 escape hatch. Phase 5 is now real; flag is misleading. Removed.
- `Simulator._has_rotating_bodies(state)` — was a short-circuit guard that never triggered in real games (planet count assumption). Replaced by always-call. Removed.
- `tests/test_sim.py::TestSimulatorPhaseStubs::test_phase_5_rotation_still_raises` — Phase 5 no longer raises. Removed.

## Notes for future-me

- **The 100% Day 5-7 result is striking.** The master design doc treats
  Day 14 ≥99% as a meaningful hurdle; we're already past it on the
  scenarios we test. Two caveats:
  1. We exclude comet-active scenarios entirely. Day 9-11 will add them.
  2. We exclude COMET_SPAWN_STEPS transitions entirely. Phase 1 spawn
     uses non-reproducible RNG; those will likely never match.
- **The step-increment bug never tripped a test.** It would have shown
  up as planet position drift, but state_diff doesn't check planet x/y.
  Could have stayed latent until something downstream depended on
  rotated positions. The fix was incidental to Phase 5 implementation
  but worth noting as a class of bug (semantic ordering errors that
  state_diff can't catch).
- **`spawned_at_step` semantics changed.** Previously fleets were spawned
  with `spawned_at_step = state.step` AFTER step was already incremented
  (so spawned_at_step = N+1). Now it's pre-increment (= N). state_diff
  doesn't check this field, but if any future logic does, it now matches
  env. Tests that explicitly assert spawned_at_step might need review;
  none currently do.
- **MCTS continuation gate (Day 14) is essentially decided.** With both
  filtered gates at 100%, the simulator is a solid foundation for MCTS
  rollouts. The remaining work (Day 9-11 comet handling, then Day 14
  unfiltered) is about coverage not correctness.
