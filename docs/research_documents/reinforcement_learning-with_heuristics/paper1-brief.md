# Paper 1 — DRLH (Deep RL Hyperheuristic for Combinatorial Optimization)

**Source PDF:** `paper1.pdf` in this directory.

## Executive summary

Kallestad et al. (EJOR 2023) propose DRLH, a PPO-trained MLP that replaces the adaptive layer of Adaptive Large Neighborhood Search (ALNS) — at every iteration the RL agent picks WHICH heuristic from a fixed pool to apply next, using problem-independent search-state features (cost deltas, temperature, no_improvement counter, last_action one-hot). DRLH outperforms ALNS, Uniform Random Selection, and tuned random selection (IRace) on CVRP/PJSP/PDP/PDPTW with the gap widening on larger instances; the pattern is directly relevant to Orbit Wars only if we first refactor v1.5G into a discrete pool of competing heuristics.

## 1. Title, authors, venue/year
"A general deep reinforcement learning hyperheuristic framework for solving combinatorial optimization problems," Jakob Kallestad, Ramin Hasibi, Ahmad Hemmati (Univ. of Bergen), Kenneth Sorensen (Univ. of Antwerp). European Journal of Operational Research 309 (2023) 446-468 (p.1).

## 2. Problem setting / domain
Selection hyperheuristics for combinatorial optimization — specifically improving the heuristic-selection layer of ALNS (Ropke & Pisinger 2006) on routing/scheduling problems: CVRP, Parallel Job Scheduling (PJSP), Pickup-and-Delivery (PDP), and PDP with Time Windows (PDPTW) (p.2, p.7). Off-topic relative to RTS/multi-agent games but on-topic for "RL-augmented heuristic" patterns.

## 3. Method
DRLH wraps the standard LNS outer loop (Algorithm 1, p.4): at each iteration t a heuristic h is sampled from policy pi(h | s_t, theta), applied to current solution x to produce x', accepted via simulated-annealing criterion exp(-|delta_E|/T) with Crama-Schyns cooling (p.5). The RL agent is an MLP with two hidden layers of size 256, learning rate 1e-5, batch 64, discount 0.5, trained with PPO (Schulman et al. 2017) for 1k iterations per episode over 5000 training instances (Tables 4, 9; p.7-8). Heuristic pool H is built (Algorithm 2, p.4) by Cartesian-producting 7 removal x 4 insertion operators plus one "Find_single_best" standalone, |H|=29 (p.4-5; Tables 1-2). The state vector (Table 3, p.6) is intentionally PROBLEM-INDEPENDENT: 12 features including reduced_cost, cost_from_min, cost, min_cost, temp, cooling-schedule alpha, no_improvement, index_step, was_changed, unseen, last_action_sign, and last_action one-hot. The reward function R^5310 (eq. 5, p.6) is: 5 if new best, 3 if just-improving, 1 if accepted by SA, 0 otherwise — a shaped per-step reward designed to avoid reward hacking. Action space = the heuristic pool itself.

## 4. Key results
On 100-instance test sets at 1k iterations DRLH beats ALNS and TRS (and URS) on essentially every problem/size, with the gap widening on larger instances (Fig. 1, p.9): CVRP-500 DRLH improves 13.54% over URS vs ALNS 1.84%; PJSP-500 5.07% vs 0.87%; PDP-100 7.34% vs 3.50%; PDPTW-300 2.97% vs 0.91%. The advantage shrinks at 25k iterations as all methods converge (Fig. 2). On the extended pool of 142 heuristics (Appendix B), DRLH degrades only ~0.02% on average while ALNS degrades substantially (Fig. 3, p.10) — DRLH learns to ignore bad heuristics. Versus the constructive Attention-Module RL of Kool et al. 2019 (Fig. 6, p.13), DRLH wins on every CVRP size including N=20 where AM cannot beat URS. The "wasted actions" analysis (Table 11, p.12) shows DRLH almost never repeats a deterministic heuristic that didn't help (~0% across most settings) while ALNS wastes 18-30% of deterministic-heuristic invocations. Ablation: alternative reward R^MC trains slower and ends worse than R^5310 at 10k iterations though wins at 1k (Tables A.1, A.2, p.13-14).

## 5. Heuristic-RL integration
This is a META-CONTROLLER / SELECTION pattern, not a policy prior. The interface is sharply defined: low-level heuristics ARE the action space. The RL agent never proposes a solution edit directly — it picks one of |H| pre-built (removal x insertion) operators and lets it run. Communication: heuristics expose nothing to the agent except their effect on the solution; the agent observes the search state via 12 generic features. The heuristic side is treated as a black-box oracle the agent learns to schedule. Authors explicitly contrast this with "macro-level" ALNS (segment-averaged scoring) — DRLH operates "micro-level," changing selection probabilities iteration-to-iteration based on current state (p.2, p.11-12). It is NOT a policy prior, NOT a reward shaper for an RL solver, NOT an action mask, NOT a search-tree pruner — it's pure operator scheduling.

## 6. Strengths / limitations (per authors)
Strengths: problem-independent state and reward generalize across 4 problem classes without per-problem feature engineering (p.6, p.13); training pipeline is the same for any new problem given heuristic operators; robust under inflated heuristic pools where ALNS collapses (p.10); per-instance overhead negligible — cost dominated by heuristics themselves (p.11, Table 10). Limitations they flag: enriching state with problem-specific features helps but "we lose some of the generality" (p.13); R^MC reward more aligned with objective but unstable to train (p.14); requires good heuristics to exist a priori — generation is out of scope.

## 7. Applicability to Orbit Wars
Conditional and not turn-key. v1.5G is currently MONOLITHIC — a single nearest-target sniper plus defense — so the paper's premise (a pool of competing heuristics) doesn't yet exist. To adopt the pattern in 1-2 weeks: factor v1.5G into 4-8 discrete decision strategies (e.g., greedy-nearest, Hungarian-assignment, defense-priority, swarm-on-weakest-enemy, hold-and-build, comet-snipe, no-op) exposed as a heuristic pool; build a 12-feature problem-independent state (own-ship-share, threat-pressure, step/EPISODE_STEPS, last_action, recent reward delta, etc.); train PPO offline against the 5-opponent sparring pool. Hard blockers: (a) Orbit Wars is a 4-player game, not single-objective optimization — the "iteration" loop becomes a turn loop where opponents move between actions, so credit assignment is much noisier than ALNS's deterministic-cost setting; (b) the per-turn 1s wall-clock is fine for a small MLP but training requires a fast simulator; (c) we'd lose the paper's biggest reported edge — large-pool robustness — because we'll start with very few heuristics. Realistic timeline: factoring + scaffolding 3-5 days, training infrastructure 3-5 days, PPO training and tuning a week+. Higher-leverage adaptations that don't require RL: use the "no-wasted-action" insight to detect and avoid repeating deterministic actions (Find_single_best analog), and use the segment-vs-iteration distinction as a critique of any future bandit-style adaptation.

## 8. What couldn't be determined
PPO hyperparameter detail beyond Table 4 (clip ratio, GAE lambda, epochs/update); exact MLP input encoding for last_action one-hot dimension when |H| is extended to 142; per-heuristic learned weights; whether DRLH was tested under wall-clock budgets vs iteration budgets (results are all iteration-indexed, which favors DRLH since it uses more compute per call than URS); convergence plots for PDPTW-300 beyond 1k iterations.
